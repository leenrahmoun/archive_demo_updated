## 1. Introduction

### 1.1 Purpose

تهدف هذه الوثيقة إلى توصيف المتطلبات الوظيفية وغير الوظيفية للنظام بصياغة رسمية تعكس السلوك الحالي المنفذ فعليًا، وليس الدراسة التحليلية الأولى فقط. تم إعدادها لتكون مرجعًا موحدًا لأصحاب القرار، محللي النظم، المطورين، المختبرين، وفريق التشغيل عند مراجعة أو تطوير النسخة الحالية.

### 1.2 Scope

يغطي النظام إدارة الأضابير الإدارية الرقمية والوثائق المرتبطة بها، مع فرض دورة عمل واضحة للوثيقة، وصلاحيات مبنية على الدور، وإتاحة قراءة ملفات PDF ومعاينتها وطباعتها ضمن حدود الرؤية المصرح بها، بالإضافة إلى سجل تدقيق شامل للعمليات الجوهرية والأحداث الأمنية.

### 1.3 Definitions / Terminology

| المصطلح | التعريف |
|---|---|
| الإضبارة | السجل الأساسي للموظف أو الحالة الإدارية، ويتضمن بيانات الهوية وموقع الحفظ الفيزيائي والوثائق المرتبطة |
| الوثيقة | مستند إداري مرتبط بإضبارة محددة ويحمل رقمًا ونوعًا واسمًا وملف PDF |
| مدخل البيانات | المستخدم المسؤول عن إنشاء الإضبارة مع أول وثيقة، وإضافة الوثائق، وتعديل مسوداته، وإرسالها للتدقيق |
| المدقق | المستخدم المسؤول عن مراجعة الوثائق ضمن نطاق التكليف المرتبط به واعتمادها أو رفضها |
| القارئ | مستخدم قراءة فقط، يرى الوثائق المعتمدة ضمن ما تسمح به قواعد الرؤية |
| المدير / Admin | المستخدم الإداري الأعلى داخل النظام التشغيلي، ويملك صلاحيات الإدارة العامة والمراجعة والاستعادة والرقابة |
| حالة الوثيقة | قيمة سير العمل الحالية للوثيقة: `draft`, `pending`, `approved`, `rejected` |
| حذف منطقي | إخفاء الوثيقة من القوائم التشغيلية دون حذفها ماديًا من قاعدة البيانات |
| استعادة | إعادة الوثيقة المحذوفة منطقيًا إلى القوائم التشغيلية دون تغيير حالتها الأساسية |
| سجل التدقيق | سجل زمني للعمليات الجوهرية والأمنية مع بيانات المنفذ والكيان والقيم قبل/بعد |

### 1.4 Intended Audience

- الإدارة المالكة للنظام
- محللو الأعمال والنظم
- فريق التطوير
- فريق الاختبار وضمان الجودة
- فريق الأمن والتدقيق الداخلي
- فريق التشغيل والدعم

## 2. Project Overview

### 2.1 وصف المشكلة

تعتمد المؤسسة على أضابير ووثائق إدارية تحتاج إلى ضبط مركزي لبياناتها ومواقع حفظها وحالات مراجعتها، مع الحاجة إلى منع الوصول غير المصرح به، وتقليل العشوائية في إدخال الوثائق، وتوثيق كل قرار أو تعديل أو اعتماد أو رفض.

### 2.2 الهدف من النظام

يهدف النظام إلى:

- ضبط إنشاء الإضبارة والوثيقة الأولى ضمن عملية واحدة غير قابلة للتجزئة
- فرض دورة عمل واضحة للوثيقة من المسودة حتى الاعتماد أو الرفض
- تطبيق رؤية مختلفة للوثائق والأضابير بحسب الدور التنظيمي
- إتاحة قراءة ملفات PDF ومعاينتها وطباعتها بصورة آمنة
- توثيق كل العمليات الجوهرية والأحداث الأمنية في سجل تدقيق قابل للبحث

### 2.3 القيمة التي يقدمها للمؤسسة

يوفر النظام قيمة تشغيلية ورقابية مباشرة من خلال:

- رفع موثوقية الأرشفة الرقمية وربطها بالإضبارة الصحيحة
- تقليل الأخطاء التشغيلية الناتجة عن الإنشاء غير المنضبط أو التعديل غير المصرح
- تسهيل المراجعة المؤسسية عبر توزيع واضح للأدوار بين الإدخال والتدقيق والقراءة والإدارة
- تعزيز المساءلة من خلال التتبع الكامل للإنشاء والتعديل والاعتماد والرفض والحذف والاستعادة ومحاولات الوصول غير المسموح
- تمكين الإدارة من متابعة مؤشرات العمل والعبء التشغيلي عبر لوحة الإدارة الحالية

## 3. التحديثات المعتمدة بعد النسخة التحليلية الأولى

تعكس النسخة الحالية من المشروع تغييرات تشغيلية مهمة مقارنةً بالتصور التحليلي الأول، وأبرزها:

- تم اعتماد ربط مستخدم `data_entry` بمدقق محدد `assigned_auditor` كجزء أساسي من منطق المراجعة.
- أصبحت رؤية المدقق للوثائق والأضابير مقيدة بنطاق المستخدمين المرتبطين به فقط، ولا تشمل مسودات ذلك النطاق.
- تمت إضافة إدارة إدارية للمستخدمين داخل النظام، بما يشمل إنشاء المستخدمين، تعديل أدوارهم، تفعيلهم أو تعطيلهم، وربط مدخلي البيانات بالمدققين.
- تمت إضافة إدارة إدارية لأنواع الوثائق، مع دعم التفعيل والتعطيل والحذف النهائي للأنواع غير المستخدمة.
- تمت إضافة الحذف المنطقي للوثائق مع صفحة مستقلة للوثائق المحذوفة منطقيًا وإجراء استعادة إداري.
- تمت إضافة لوحة إدارة تشغيلية تعرض ملخصات، مؤشرات، نشاطًا حديثًا، وتحليلات مبنية على البيانات الفعلية.
- تمت تقوية المصادقة عبر JWT مع تدوير رمز التحديث ورفض الحسابات غير النشطة وتسجيل أحداث الدخول والخروج وفشل التحديث.
- تمت إضافة قراءة ملف PDF داخل الصفحة، وفتحه في صفحة جديدة، وطباعته، مع فرض نفس قيود الوصول المعتمدة على عرض تفاصيل الوثيقة.
- في السياسة التشغيلية المنفذة حاليًا، الوثيقة المعتمدة لا تعدل من أي دور، بما في ذلك المدير، ولم تُنفذ آلية استثنائية لتحرير وثيقة معتمدة في النسخة الحالية.
- حالة `is_archived` للإضبارة معتمدة وظيفيًا كقيد يمنع إضافة وثائق جديدة إليها، إلا أن مسار أرشفة/إلغاء أرشفة الإضبارة لم يُطرح كواجهة تشغيلية مكتملة بعد.

## 4. System Scope

### 4.1 ما الذي يشمله النظام

يشمل النظام في نسخته الحالية:

- المصادقة المعتمدة على JWT وتحديث الجلسة وتسجيل الخروج
- إدارة الأضابير
- إنشاء الإضبارة مع أول وثيقة في معاملة واحدة
- إدارة الوثائق وإرفاق ملفات PDF
- سير مراجعة الوثيقة: إرسال، اعتماد، رفض، إعادة إرسال
- الحذف المنطقي والاستعادة
- قوائم البحث والتصفية والترتيب والترقيم الصفحي
- قائمة مراجعة خاصة بالمدققين والإدارة للوثائق المعلقة
- سجل تدقيق إداري للعمليات الجوهرية والأحداث الأمنية
- إدارة المستخدمين للأدوار التشغيلية
- إدارة أنواع الوثائق
- لوحة إدارة ومؤشرات تشغيلية
- صلاحيات وصول ورؤية مبنية على الدور ونطاق التكليف

### 4.2 ما الذي لا يشمله النظام حاليًا / المؤجل للإصدارات اللاحقة

- مسار تشغيل كامل لأرشفة الإضبارة وإلغاء أرشفتها من واجهة المستخدم
- إدارة إدارية كاملة للمحافظات المرجعية
- استيراد جماعي للوثائق أو الأضابير
- تكامل مع أنظمة خارجية
- محرك تقارير قابل للتخصيص أو تصدير تحليلات موسع
- إدارة نسخ متعددة للوثيقة الواحدة
- تحرير استثنائي لمحتوى وثيقة معتمدة بعد اعتمادها
- أتمتة النشر والإدارة التشغيلية الإنتاجية الكاملة

## 5. User Roles

### 5.1 Admin

المسؤول الإداري الأعلى داخل النظام التشغيلي، ويملك صلاحيات الإدارة والرقابة، ويمكنه:

- رؤية جميع الأضابير والوثائق غير المحذوفة منطقيًا
- إنشاء الأضابير والوثائق
- تعديل الوثائق في حالتي `draft` و`rejected`
- اعتماد أو رفض أي وثيقة `pending` حتى خارج نطاق تكليف المدققين
- حذف الوثائق منطقيًا
- عرض قائمة الوثائق المحذوفة منطقيًا واستعادتها
- إدارة المستخدمين
- إدارة أنواع الوثائق
- عرض سجل التدقيق
- عرض لوحة الإدارة

### 5.2 Data Entry

المستخدم التنفيذي المسؤول عن الإدخال الأولي، ويمكنه:

- إنشاء الإضبارة مع أول وثيقة
- إضافة وثائق إلى الأضابير التي أنشأها فقط
- تعديل وثائقه الخاصة إذا كانت `draft` أو `rejected`
- استبدال ملف PDF لوثائقه الخاصة في حالتي `draft` و`rejected`
- إرسال الوثيقة للتدقيق أو إعادة إرسال الوثيقة المرفوضة
- حذف وثيقته منطقيًا إذا كانت `draft` فقط
- رؤية أضابيره ووثائقه فقط

### 5.3 Auditor / Reviewer

المستخدم المسؤول عن المراجعة ضمن نطاق التكليف، ويمكنه:

- رؤية الوثائق `pending` و`rejected` و`approved` الخاصة بمدخلي البيانات المرتبطين به فقط
- مراجعة الوثائق المعلقة ضمن قائمة مراجعة خاصة
- اعتماد أو رفض الوثيقة `pending` ضمن نطاقه فقط
- قراءة ملفات PDF ضمن نطاقه

ولا يمكنه:

- إنشاء أضابير أو وثائق
- تعديل محتوى الوثيقة
- رؤية المسودات
- استعادة المحذوف منطقيًا
- الوصول إلى سجل التدقيق الإداري

### 5.4 Reader

المستخدم ذو الوصول القرائي فقط، ويمكنه:

- رؤية الأضابير التي تحتوي وثائق معتمدة مرئية له
- رؤية الوثائق `approved` فقط
- قراءة ملفات PDF وفتحها وطباعتها ضمن هذا النطاق

ولا يمكنه:

- إنشاء أو تعديل أو حذف أو إرسال أو اعتماد أو رفض
- الوصول إلى سجل التدقيق أو إدارة المستخدمين أو لوحة الإدارة أو قائمة المحذوفات

## 6. Functional Requirements

### FR-01 Authentication and Session Management

- FR-01.1 يجب أن يدعم النظام تسجيل الدخول باستخدام اسم المستخدم وكلمة المرور.
- FR-01.2 يجب أن يعتمد النظام على JWT بزوج من الرموز: `access` و`refresh`.
- FR-01.3 يجب أن يدعم النظام تحديث الجلسة باستخدام `refresh token`.
- FR-01.4 يجب أن يطبق النظام تدوير `refresh token` وإدخاله في قائمة المنع بعد الاستبدال.
- FR-01.5 يجب أن يرفض النظام الحسابات غير النشطة عند تسجيل الدخول، وعند تحديث الجلسة، وعند محاولة الوصول إلى أي واجهة محمية.
- FR-01.6 يجب أن يتيح النظام للمستخدم معرفة ملفه الشخصي الحالي ودوره التنظيمي والمدقق المرتبط به إن وجد.
- FR-01.7 يجب أن يسجل النظام أحداث الدخول الناجح، وفشل الدخول، وتسجيل الخروج، وفشل تحديث الجلسة في سجل التدقيق.

### FR-02 Dossier Management

- FR-02.1 يجب أن يدعم النظام إنشاء الإضبارة مع أول وثيقة ضمن طلب واحد ومعاملة واحدة.
- FR-02.2 يجب أن يمنع النظام إنشاء إضبارة فارغة.
- FR-02.3 يجب أن يحتوي سجل الإضبارة على رقم الملف، الاسم الكامل، الرقم الوطني، الرقم الذاتي، الجنسية/المحافظة، وموقع الحفظ الفيزيائي.
- FR-02.4 يجب أن يدعم النظام اختلاف منطق التحقق بين السوري وغير السوري.
- FR-02.5 يجب أن يتيح النظام عرض قائمة الأضابير مع البحث والتصفية والترتيب والترقيم الصفحي.
- FR-02.6 يجب أن يطبق النظام رؤية الأضابير حسب الدور ونطاق الصلاحية.
- FR-02.7 يجب أن يعرض النظام داخل تفاصيل الإضبارة الوثائق المسموح للمستخدم الحالي برؤيتها فقط.
- FR-02.8 يجب أن يمنع النظام إضافة وثائق إلى إضبارة موسومة كـ `is_archived`.

### FR-03 Document Management

- FR-03.1 يجب أن يدعم النظام إضافة وثيقة جديدة إلى إضبارة موجودة.
- FR-03.2 يجب أن تكون كل وثيقة مرتبطة بإضبارة واحدة ونوع وثيقة واحد.
- FR-03.3 يجب أن تحفظ الوثيقة مع رقم وثيقة واسم وثيقة وملاحظات اختيارية وملف PDF.
- FR-03.4 يجب أن يبدأ إنشاء الوثيقة بحالة `draft`.
- FR-03.5 يجب أن يمنع النظام تعديل الوثيقة إلا إذا كانت `draft` أو `rejected`.
- FR-03.6 يجب أن يسمح النظام للمدير بتعديل الوثائق غير المحذوفة في حالتي `draft` و`rejected`.
- FR-03.7 يجب أن يسمح النظام لمدخل البيانات بتعديل وثائقه فقط في حالتي `draft` و`rejected`.
- FR-03.8 يجب أن يسمح النظام باستبدال ملف PDF لمدخل البيانات فقط، وعلى وثائقه الخاصة فقط، وفي حالتي `draft` و`rejected`.
- FR-03.9 يجب أن يرفض النظام أي تعديل على وثيقة `pending` أو `approved`.
- FR-03.10 يجب أن يدعم النظام الحذف المنطقي للوثيقة دون حذف سجلها من قاعدة البيانات.
- FR-03.11 يجب أن يدعم النظام عرض قائمة منفصلة للوثائق المحذوفة منطقيًا للإدارة فقط.
- FR-03.12 يجب أن يدعم النظام استعادة الوثيقة المحذوفة منطقيًا للإدارة فقط.
- FR-03.13 يجب أن يمنع النظام وجود وثيقتين نشطتين داخل الإضبارة نفسها تحملان نفس نوع الوثيقة ونفس رقم الوثيقة.

### FR-04 PDF File Validation, Storage, Viewing, Preview, and Print

- FR-04.1 يجب أن يقبل النظام ملفات PDF فقط.
- FR-04.2 يجب أن يفرض النظام حدًا أدنى لحجم الملف مقداره 100KB وحدًا أعلى مقداره 15MB.
- FR-04.3 يجب أن يطبق التحقق على الملف في الواجهة الأمامية والخلفية.
- FR-04.4 يجب أن يخزن النظام الملف تحت المسار الداخلي الخاص بالرفع المرتبط بالإضبارة.
- FR-04.5 يجب ألا يُتاح الوصول إلى الملف مباشرةً إلا عبر واجهة backend تطبق قواعد الصلاحية.
- FR-04.6 يجب أن يدعم النظام قراءة ملف PDF داخل الصفحة.
- FR-04.7 يجب أن يدعم النظام فتح ملف PDF في صفحة جديدة.
- FR-04.8 يجب أن يدعم النظام طباعة ملف PDF ضمن النطاق المسموح للمستخدم.
- FR-04.9 يجب أن تكون صلاحيات قراءة/فتح/طباعة PDF متطابقة مع صلاحيات رؤية تفاصيل الوثيقة نفسها.

### FR-05 Review / Approval Workflow

- FR-05.1 يجب أن تمر الوثيقة بحالات `draft`, `pending`, `approved`, `rejected` فقط.
- FR-05.2 يجب أن يسمح النظام بإرسال الوثيقة من `draft` إلى `pending`.
- FR-05.3 يجب أن يسمح النظام بإعادة إرسال الوثيقة من `rejected` إلى `pending`.
- FR-05.4 يجب أن يسمح النظام باعتماد الوثيقة من `pending` إلى `approved`.
- FR-05.5 يجب أن يسمح النظام برفض الوثيقة من `pending` إلى `rejected`.
- FR-05.6 يجب أن يشترط النظام سببًا للرفض عند الانتقال إلى `rejected`.
- FR-05.7 يجب أن يربط النظام قرار الاعتماد أو الرفض بالمستخدم المراجع وتاريخ المراجعة.
- FR-05.8 يجب أن يتيح النظام قائمة مراجعة خاصة بالوثائق `pending` للمدققين والمدير.
- FR-05.9 يجب أن يقيد اعتماد/رفض المدقق بوثائق مدخلي البيانات المرتبطين به فقط.
- FR-05.10 يجب أن يسمح للمدير باعتماد أو رفض أي وثيقة `pending` ضمن النظام.

### FR-06 Search, Filtering, Ordering, and Pagination

- FR-06.1 يجب أن يدعم النظام البحث في الأضابير برقم الإضبارة والاسم والرقم الوطني والرقم الذاتي.
- FR-06.2 يجب أن يدعم النظام البحث في الوثائق برقم الوثيقة واسمها ورقم الإضبارة والمسار.
- FR-06.3 يجب أن يدعم النظام تصفية الأضابير بحسب المحافظة والحالة الأرشيفية والمُنشئ ضمن النطاق المسموح.
- FR-06.4 يجب أن يدعم النظام تصفية الوثائق بحسب الحالة ونوع الوثيقة والإضبارة والمنشئ والمراجع، وبحسب الحذف أو الحذف المنطقي حيثما ينطبق.
- FR-06.5 يجب أن يدعم النظام الترتيب في القوائم الرئيسية.
- FR-06.6 يجب أن يدعم النظام الترقيم الصفحي في القوائم.
- FR-06.7 يجب ألا تسمح عمليات البحث أو التصفية أو الترتيب بتوسيع الرؤية خارج النطاق المسموح للمستخدم.

### FR-07 Audit Log

- FR-07.1 يجب أن يسجل النظام إنشاء الوثيقة وإنشاء الإضبارة والتعديل والإرسال والاعتماد والرفض واستبدال الملف والحذف والاستعادة.
- FR-07.2 يجب أن يسجل النظام أحداث الدخول والخروج وفشل الدخول وفشل تحديث الجلسة ومحاولات الوصول المرفوضة في سياق العمليات الحرجة.
- FR-07.3 يجب أن يحتفظ السجل بالمنفذ، والإجراء، والكيان، والمعرف، والقيم قبل/بعد، وتاريخ التنفيذ، وعنوان الشبكة عند توافره.
- FR-07.4 يجب أن يتيح النظام البحث والتصفية والاطلاع التفصيلي على سجل التدقيق للإدارة فقط.
- FR-07.5 يجب أن يعرض النظام أوصافًا بشرية مفهومة للحدث والكيان وملخص التغيير.

### FR-08 User Management

- FR-08.1 يجب أن يتيح النظام للإدارة إنشاء المستخدمين التشغيليين وتعديلهم وحذفهم.
- FR-08.2 يجب أن يدعم النظام الأدوار: `admin`, `data_entry`, `auditor`, `reader`.
- FR-08.3 يجب أن يشترط النظام ربط مستخدم `data_entry` بمدقق `auditor` عند الإنشاء.
- FR-08.4 يجب أن يمنع النظام ربط أي دور آخر بمدقق معيّن.
- FR-08.5 يجب أن يتيح النظام تعديل كلمة المرور اختياريًا عند تعديل المستخدم.
- FR-08.6 يجب أن يمنع النظام تغيير دور مدخل البيانات إذا كان يملك وثائق غير نهائية (`draft`, `rejected`, `pending`).
- FR-08.7 يجب أن يمنع النظام تغيير دور المدقق إذا كان ما يزال مرتبطًا بمدخلي بيانات.
- FR-08.8 يجب أن يسجل النظام تغييرات الدور وتغييرات المدقق المرتبط في سجل التدقيق.
- FR-08.9 يجب أن يخفي النظام الحسابات الطارئة/فائقة الصلاحية من القوائم التشغيلية الاعتيادية لإدارة المستخدمين.

### FR-09 Document Types and Reference Data

- FR-09.1 يجب أن يقدم النظام قائمة أنواع وثائق فعالة للاختيار أثناء إنشاء أو تعديل الوثائق.
- FR-09.2 يجب أن يقدم النظام قائمة المحافظات الفعالة للاختيار أثناء إنشاء الإضبارة.
- FR-09.3 يجب أن يتيح النظام للإدارة إنشاء نوع وثيقة جديد وتعديله وتفعيله وتعطيله.
- FR-09.4 يجب أن يمنع النظام اختيار نوع وثيقة غير فعال في الوثائق الجديدة.
- FR-09.5 يجب أن يسمح النظام بالإبقاء على نوع وثيقة غير فعال إذا كان النوع نفسه مرتبطًا أصلًا بوثيقة قائمة ويجري فقط تعديل بقية الحقول.
- FR-09.6 يجب أن يمنع النظام حذف نوع وثيقة مستخدم في وثائق قائمة.
- FR-09.7 يجب أن يسجل النظام إنشاء أنواع الوثائق وتعديلها وتعطيلها وحذفها في سجل التدقيق.

### FR-10 Administrative Dashboard and Operational Monitoring

- FR-10.1 يجب أن يتيح النظام للإدارة لوحة مؤشرات تشغيلية.
- FR-10.2 يجب أن تعرض اللوحة ملخصًا لعدد الأضابير والوثائق حسب الحالة وعدد المستخدمين النشطين والوثائق المحذوفة منطقيًا.
- FR-10.3 يجب أن تعرض اللوحة مؤشرات لسير العمل مثل عدد الوثائق المعلقة والوثائق المرفوضة بانتظار التصحيح.
- FR-10.4 يجب أن تعرض اللوحة مؤشرات أداء لمدخلي البيانات والمدققين.
- FR-10.5 يجب أن تعرض اللوحة نشاطًا حديثًا للوثائق وسجل التدقيق.

### FR-11 Permissions and Visibility Rules

- FR-11.1 يجب أن تكون كل صلاحية مطبقة على مستوى backend، وليس على مستوى الواجهة فقط.
- FR-11.2 يجب أن يرى `data_entry` أضابيره ووثائقه فقط.
- FR-11.3 يجب أن يرى `auditor` وثائق النطاق المرتبط به فقط، وبالحالات `pending`, `rejected`, `approved`.
- FR-11.4 يجب ألا يرى `auditor` أي وثيقة `draft`.
- FR-11.5 يجب أن يرى `reader` الوثائق `approved` فقط.
- FR-11.6 يجب أن يرى `reader` فقط الأضابير التي تحتوي وثائق معتمدة مرئية له.
- FR-11.7 يجب أن تُستبعد الوثائق المحذوفة منطقيًا من القوائم والتفاصيل القياسية.
- FR-11.8 يجب ألا يطّلع على الوثائق المحذوفة منطقيًا إلا المدير عبر المسار المخصص لذلك.
- FR-11.9 يجب أن يسجل النظام محاولة تنفيذ إجراء سير عمل خارج النطاق المسموح كحدث أمني في سجل التدقيق.

## 7. Non-Functional Requirements

### NFR-01 Security

- يجب فرض المصادقة على جميع الواجهات المحمية.
- يجب تطبيق التحقق من نشاط الحساب على جميع مسارات الجلسة.
- يجب أن يكون الوصول إلى ملفات PDF عبر backend فقط مع التحقق من الدور والنطاق.
- يجب ألا تعتمد الصلاحيات على الواجهة فقط.
- يجب أن تُمنع رؤية البيانات خارج النطاق حتى عند معرفة المعرّفات مباشرة.

### NFR-02 Performance

- يجب أن تدعم القوائم الرئيسية البحث والتصفية والترتيب مع أداء مقبول على أحجام البيانات التشغيلية.
- يجب أن تستفيد البيانات الأساسية من الفهارس المعرفة على الحقول المرجعية مثل رقم الإضبارة والرقم الوطني والحالة.
- يجب أن تكون نتائج القوائم مترقمة صفحيًا لتجنب تحميل غير محدود للبيانات.

### NFR-03 Usability

- يجب أن تكون الواجهة عربية وواضحة وقابلة للاستخدام المؤسسي اليومي.
- يجب أن تعرض الواجهة رسائل خطأ مفهومة للمستخدم.
- يجب أن تكون حالات الوثيقة وإجراءاتها مرئية بوضوح داخل التفاصيل والقوائم.
- يجب أن تكون إجراءات PDF متاحة للمستخدم وفق صلاحياته دون الحاجة لتنزيل يدوي خارج النظام.

### NFR-04 Reliability

- يجب أن تكون عملية إنشاء الإضبارة مع أول وثيقة ذرّية بالكامل.
- يجب ألا يؤدي فشل حفظ الوثيقة الأولى أو الملف إلى بقاء إضبارة ناقصة.
- يجب تنظيف الملف المرفوع إذا فشل حفظ السجل المنطقي المرتبط به.

### NFR-05 Maintainability

- يجب أن تبقى قواعد الأعمال الحرجة معزولة في طبقات خدمة وصلاحية قابلة للاختبار.
- يجب أن تكون الرسائل والحالات والصلاحيات قابلة للتتبع في الكود والاختبارات.
- يجب دعم تطوير محلي عبر PostgreSQL بوصفه الخيار الأساسي، مع SQLite كخيار fallback للتطوير المحلي المحدود فقط.

### NFR-06 Auditability

- يجب أن تكون العمليات الجوهرية قابلة للتتبع بوضوح من خلال سجل التدقيق.
- يجب أن يشمل التتبع القيم القديمة والجديدة متى كان ذلك منطقيًا.
- يجب أن يكون سجل التدقيق قابلًا للبحث والتصفية والإظهار البشري داخل واجهة الإدارة.

### NFR-07 Scalability

- يجب أن يبنى النظام على واجهات API قابلة للتوسع لتدعم زيادة عدد المستخدمين والوثائق.
- يجب أن تبقى قواعد الرؤية قابلة للتطبيق على الاستعلامات نفسها، لا على النتائج بعد جلبها.

## 8. Business Rules

| الرمز | القاعدة |
|---|---|
| BR-01 | لا يجوز إنشاء إضبارة فارغة. |
| BR-02 | يجب إنشاء الإضبارة مع أول وثيقة ضمن عملية واحدة ومعاملة واحدة. |
| BR-03 | حالات الوثيقة المعتمدة حاليًا هي فقط: `draft`, `pending`, `approved`, `rejected`. |
| BR-04 | الوثيقة `pending` لا تعدل ولا يعاد إرسالها قبل صدور قرار مراجعة. |
| BR-05 | الوثيقة `rejected` يمكن تصحيحها وتعديلها ثم إعادة إرسالها. |
| BR-06 | رفض الوثيقة يتطلب سبب رفض إلزاميًا. |
| BR-07 | الوثيقة `approved` تعد وثيقة نهائية للنسخة الحالية، ولا توجد آلية تحرير محتوى معتمدة لها، حتى إداريًا. |
| BR-08 | القارئ يرى الوثائق المعتمدة فقط. |
| BR-09 | مدخل البيانات يتعامل فقط مع أضابيره ووثائقه ضمن الصلاحيات المعتمدة. |
| BR-10 | مدخل البيانات لا يستطيع إرسال الوثيقة للمراجعة ما لم يكن مرتبطًا بمدقق محدد. |
| BR-11 | المدقق يرى فقط وثائق مدخلي البيانات المرتبطين به، ولا يرى المسودات. |
| BR-12 | المدير يستطيع مراجعة أي وثيقة `pending` داخل النظام، ولو كانت خارج نطاق تكليف المدققين. |
| BR-13 | الحذف في النسخة الحالية حذف منطقي فقط، ولا تظهر الوثيقة المحذوفة في الاستعلامات القياسية. |
| BR-14 | استعادة الوثيقة المحذوفة منطقيًا صلاحية إدارية فقط في السلوك المنفذ حاليًا. |
| BR-15 | مدخل البيانات يستطيع حذف وثيقته منطقيًا إذا كانت `draft` فقط. |
| BR-16 | المدير يستطيع حذف أي وثيقة غير محذوفة منطقيًا. |
| BR-17 | نوع الوثيقة غير الفعال لا يجوز اختياره في وثيقة جديدة أو عند التبديل إلى نوع جديد غير فعال. |
| BR-18 | تعطيل نوع الوثيقة لا يجب أن يكسر عرض الوثائق القديمة المرتبطة به. |
| BR-19 | لا يمكن تغيير دور مدخل البيانات إذا كان لديه وثائق غير نهائية. |
| BR-20 | لا يمكن تغيير دور المدقق إذا كان لا يزال مرتبطًا بمدخلي بيانات. |
| BR-21 | الإضبارة المؤرشفة (`is_archived`) لا تقبل إضافة وثائق جديدة. |
| BR-22 | العمليات الجوهرية والأحداث الأمنية يجب أن تسجل في Audit Log. |
| BR-23 | لا يجوز تكرار وثيقة نشطة داخل الإضبارة نفسها بنفس `(doc_type + doc_number)`، ويستثنى من ذلك السجل المحذوف منطقيًا. |
| BR-24 | لا يجوز إنشاء نوعي وثائق متطابقين اسمًا بعد التطبيع العربي المستخدم في إدارة الأنواع. |

## 9. Workflow / Document Lifecycle

| المرحلة | الوصف | المنفذ المسموح | الناتج |
|---|---|---|---|
| إنشاء إضبارة مع أول وثيقة | إدخال بيانات الإضبارة والوثيقة الأولى وملف PDF في عملية واحدة | `admin`, `data_entry` | إنشاء الإضبارة والوثيقة الأولى بحالة `draft` |
| إنشاء وثيقة إضافية | إضافة وثيقة جديدة إلى إضبارة موجودة وغير مؤرشفة | `admin`, `data_entry` | وثيقة جديدة بحالة `draft` |
| حفظ كمسودة | حفظ أو تعديل بيانات وثيقة غير نهائية | `admin`, `data_entry` وفق القيود | وثيقة تبقى `draft` أو `rejected` |
| إرسال للتدقيق | تحويل الوثيقة من `draft` أو `rejected` إلى `pending` | `admin`, أو `data_entry` المالك لوثيقته | وثيقة قيد المراجعة |
| اعتماد | اتخاذ قرار قبول نهائي لوثيقة معلقة | `auditor` ضمن النطاق، أو `admin` | وثيقة `approved` مع المراجع والتاريخ |
| رفض | اتخاذ قرار رفض لوثيقة معلقة مع سبب إلزامي | `auditor` ضمن النطاق، أو `admin` | وثيقة `rejected` مع سبب الرفض |
| تصحيح بعد الرفض | تعديل البيانات أو استبدال ملف PDF ثم إعادة الإرسال | `admin`, أو `data_entry` المالك | العودة إلى `pending` |
| حذف منطقي | إخراج الوثيقة من القوائم التشغيلية | `admin`، أو `data_entry` لنسخة `draft` الخاصة به | `is_deleted = true` |
| استعادة | إعادة الوثيقة المحذوفة منطقيًا للقوائم | `admin` فقط فعليًا | `is_deleted = false` مع بقاء الحالة الأصلية |
| عرض / قراءة / طباعة | قراءة تفاصيل الوثيقة وملف PDF ومعاينته وطباعته | حسب الدور ونطاق الرؤية | وصول مقيد بالسياسات |

### 9.1 التسلسل التشغيلي المعتمد

1. إنشاء الإضبارة مع أول وثيقة.
2. حفظ الوثيقة الأولى كمسودة.
3. إرسال الوثيقة للمراجعة.
4. يقوم المدقق أو المدير بالاعتماد أو الرفض.
5. إذا رُفضت الوثيقة، تعدل ثم يعاد إرسالها.
6. إذا اعتُمدت الوثيقة، تصبح للقراءة فقط ضمن النسخة الحالية.
7. يمكن حذف الوثيقة منطقيًا وفق الصلاحيات المعتمدة.
8. يمكن للإدارة استعادة الوثيقة المحذوفة منطقيًا.

### 10.1 ملاحظات تنفيذية على الصلاحيات

- المدقق لا يرى المسودات مطلقًا.
- مدخل البيانات لا يستطيع الوصول إلى وثائق أو أضابير مستخدم آخر حتى لو عرف المعرّف المباشر.
- الوثائق المحذوفة منطقيًا لا تظهر في المسارات القياسية ولا في تفاصيل الوثيقة دون `include_deleted`، وهذا الوصول مخصص فعليًا للإدارة فقط.

## 11. Assumptions and Constraints

- يعتمد النظام على واجهة ويب عربية للاستخدام الداخلي.
- يعتمد النظام على ملفات PDF فقط بوصفها الصيغة الرقمية الرسمية للوثائق.
- يعتمد حفظ الملفات على المسار الداخلي الخاص بالرفع ضمن بيئة backend، مع الوصول عبر واجهة محكومة بالصلاحيات.
- يعتمد النظام على صلاحيات مبنية على الدور، وعلى نطاق إضافي قائم على علاقة `data_entry` مع `assigned_auditor`.
- يعتمد النظام على سجل تدقيق بوصفه مكونًا أساسيًا وليس اختياريًا.
- تعتمد قاعدة البيانات الأساسية المقصودة على PostgreSQL، مع وجود SQLite كخيار تطوير محلي محدود فقط.
- تعتمد أنواع الوثائق والمحافظات على بيانات مرجعية داخلية، مع توفر إدارة تشغيلية لأنواع الوثائق فقط في النسخة الحالية.
- توجد حالة أرشفة للإضبارة (`is_archived`) مستخدمة كقيد تشغيلي، لكن لا توجد حتى الآن واجهة متكاملة لإدارتها من المستخدم النهائي.
- تُخفى الحسابات الفائقة/الطارئة من القوائم التشغيلية الاعتيادية لإدارة المستخدمين وسجل التدقيق الاعتيادي.

## 12. Acceptance Criteria

| الرمز | معيار القبول |
|---|---|
| AC-01 | عند محاولة إنشاء إضبارة دون وثيقة أولى، يجب أن يرفض النظام الطلب وألا ينشئ أي سجل. |
| AC-02 | عند فشل إنشاء الوثيقة الأولى أو ملفها، يجب أن تتراجع عملية إنشاء الإضبارة بالكامل وألا يبقى أي ملف مرفوع غير مرتبط. |
| AC-03 | يجب أن تبدأ كل وثيقة جديدة بحالة `draft`. |
| AC-04 | يجب ألا يسمح النظام بتعديل وثيقة `pending` أو `approved`. |
| AC-05 | يجب ألا يسمح النظام برفض وثيقة دون `rejection_reason`. |
| AC-06 | يجب أن تعود الوثيقة `rejected` إلى `pending` عند إعادة إرسالها بعد التصحيح. |
| AC-07 | يجب أن يمنع النظام مدخل البيانات من إرسال وثيقته إذا لم يكن مرتبطًا بمدقق. |
| AC-08 | يجب أن يرى `reader` الوثائق المعتمدة فقط، ويجب ألا تظهر له أي وثيقة `draft` أو `pending` أو `rejected`. |
| AC-09 | يجب أن يرى `auditor` فقط وثائق النطاق المرتبط به، وبالحالات `pending`, `rejected`, `approved` دون `draft`. |
| AC-10 | يجب أن تستبعد الوثائق المحذوفة منطقيًا من القوائم القياسية، وأن تظهر فقط في قائمة المحذوفات الخاصة بالإدارة. |
| AC-11 | يجب أن يقتصر إجراء الاستعادة على الإدارة فعليًا. |
| AC-12 | يجب أن تطبق صلاحيات الوصول إلى ملف PDF بنفس قواعد صلاحيات رؤية الوثيقة. |
| AC-13 | يجب أن تسجل العمليات الجوهرية والأحداث الأمنية في Audit Log مع بيانات المنفذ والكيان والزمن. |
| AC-14 | يجب ألا يقبل النظام اختيار نوع وثيقة غير فعال في وثيقة جديدة. |
| AC-15 | يجب أن يمنع النظام تغيير دور مدخل البيانات إذا كان لديه وثائق غير نهائية، وأن يمنع تغيير دور المدقق إذا كان مرتبطًا بمدخلي بيانات. |
| AC-16 | يجب أن يرفض النظام إنشاء أو إبقاء وثيقتين نشطتين في الإضبارة نفسها بنفس نوع الوثيقة ورقمها. |

## 13. Future Considerations

هذا القسم خارج نطاق النسخة الحالية، ويذكر فقط كاتجاهات مستقبلية محتملة:

- واجهة متكاملة لأرشفة الإضبارة وإلغاء أرشفتها
- إدارة إدارية للمحافظات المرجعية
- استيراد جماعي للوثائق أو الأضابير
- تقارير وتصدير تشغيلي موسع
- تكاملات خارجية أو أرشفة مؤسسية أوسع
- آلية مضبوطة لتعديل استثنائي لوثيقة معتمدة عند الحاجة التنظيمية
